# 文档规范 (Documentation Specifications)

本文档规定了项目文档的编写、组织和维护标准，确保文档资产的可用性和专业性。

## 1. 文档组织结构

所有文档应存放在 `docs/` 目录下，并遵循以下分类：

-   `docs/plan/`: 项目规划、待办事项、路线图。
-   `docs/standards/`: 各种规范、标准文档。
-   `docs/design/`: 架构设计、数据库设计、UI 设计、接口设计。
-   `docs/api/`: API 接口参考手册。
-   `docs/usage/`: 面向用户的安装、使用说明。
-   `docs/deployment/`: 针对不同环境的部署指南。
-   `docs/development/`: 针对开发者的起步引导和深度解析。

## 2. 编写语言与格式

-   **格式**: 统一使用 Markdown (`.md`)。
-   **语言**: 优先使用**中文**。技术专有名词可保留英文原文。
-   **中英文混排**: 建议在中文与英文/数字之间增加一个半角空格（如：“使用 TypeScript 编写”）。

## 3. 维护规范

-   **同步更新**: 文档即代码。当代码逻辑发生重大变更时，相关的文档必须同步更新。
-   **禁止虚构**: 仅记录已确定的计划或已实现的逻辑。未经证实的构想应标注为“提案”或“待定”。
-   **可视化**:
    -   复杂逻辑优先使用 Mermaid 语法绘制流程图或时序图。
    -   重要界面设计应附带截图或设计稿链接。

## 4. 风格指南

-   **标题**: 使用合理的标题等级 (`#`, `##`, `###`)。
-   **列表**: 复杂步骤使用有序列表，功能特性使用无序列表。
-   **代码块**: 必须指明代码语言类型（如：```typescript）。
-   **交叉引用**: 在文档内部引用其他文件时，使用相对路径（如：`[开发规范](../standards/development.md)`）。
